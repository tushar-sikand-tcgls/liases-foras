```
The "Attributes" list above contains the EXACT attribute names that were resolved from the query.
YOU MUST USE THESE EXACT NAMES IN YOUR QUERY PLAN.
DO NOT substitute, rename, or use similar attribute names.

Example:
  If Attributes: ["Sellout Efficiency"]
  → Use "Sellout Efficiency" in your query plan
  → DO NOT use "Sold %", "Absorption Rate", or any other metric
```

But the LLM is STILL generating:
```python

```

**Why:** The LLM is treating the query as a "project overview" query and deciding to fetch generic metadata instead of respecting the resolved attributes list.

## Recommended Fix: Option B (Post-Processing Validation)

Add validation logic in `kg_query_planner_node.py` after LLM generates the plan:

```python
# After LLM generates query_plan
resolved_attributes = planning_context['attributes']
if resolved_attributes:
    # Check if any step in the plan uses the resolved attributes
    plan_uses_resolved_attrs = False
    for step in query_plan:
        step_attributes = step.get('attributes', [])
        if any(attr in resolved_attributes for attr in step_attributes):
            plan_uses_resolved_attrs = True
            break

    # If plan doesn't use resolved attributes, force inject them
    if not plan_uses_resolved_attrs:
        print(f"⚠️  LLM generated plan without resolved attributes - auto-correcting...")
        # Replace attributes in fetch actions with resolved attributes
        for step in query_plan:
            if step.get('action') == 'fetch':
                step['attributes'] = resolved_attributes
                step['projects'] = planning_context['projects']
                print(f"  → Injected: {resolved_attributes}")
```

This is a **defensive programming** approach that ensures the system works correctly even if the LLM makes mistakes.
